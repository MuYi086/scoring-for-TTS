## 问题 1: transformers 版本不兼容

**现象**: 安装 qwen-tts 后，transformers 从 4.51.3 被升级到 4.57.3，导致 CosyVoice3 输出叽里咕噜不可理解的语音。

**原因**: CosyVoice3 的 LLM 核心组件 `Qwen2Encoder`（cosyvoice/llm/llm.py:226）直接使用 `Qwen2ForCausalLM.from_pretrained()`，transformers 4.57.3 改变了 Qwen2 模型的内部行为，导致生成的 speech tokens 全部错误。

**表现**:
- transformers 4.57.3: 某些分段仅 0.0s~0.5s，输出为乱码
- transformers 4.51.3: 分段时长正常（5s~30s），输出为可理解中文

**临时解决**: 降级 transformers 到 4.51.3（但会破坏 qwen-tts，qwen-tts 锁定 transformers==4.57.3）

**建议**: 使用独立 conda 环境分别运行两个 TTS 模型。

## 问题 2: 推理状态污染（核心问题）

**现象**: 同一个 CosyVoice3 实例，第一次 `inference_zero_shot()` 调用输出正确，后续调用输出错误内容。

**复现步骤**:
```python
model = CosyVoice3(model_dir)
# 第一次调用: 正确 (6.8s)
for r in model.inference_zero_shot("尊敬的各位听众朋友，大家好。", prompt_text, sample_wav, stream=False):
    ...
# 第二次调用: 错误 (0.4s)
for r in model.inference_zero_shot("尊敬的各位听众朋友，大家好。", prompt_text, sample_wav, stream=False):
    ...
```

**测试结果**:
- call 0: 6.8s（正确）
- call 1: 0.4s（错误）
- call 2: 3.1s（错误）

**排查过程**:
- 参考音频特征提取一致（token_len=206, feat_len=411, emb_norm=12.4671）✅
- 模型处于 eval 模式 ✅
- gc.collect() + torch.cuda.empty_cache() 无效
- 重建 ONNX speech_tokenizer_session 无效
- 每次创建新模型实例可解决（但每段多 ~5s 加载开销）

**可能原因**: Qwen2ForCausalLM 或 flow decoder 在推理后残留内部状态（如 past_key_values 缓存未正确清理），影响后续推理。

## 问题 3: 文本预处理改变内容

**现象**: `text_frontend=True`（默认）时，wetext 文本规范化 + split_paragraph 分段会导致输出内容与原始文本不一致。

**表现**:
- `text_frontend=False`: 输出时长 11.2s，内容更接近原文
- `text_frontend=True`: 输出时长 4.6s，内容偏离原文

**原因**: wetext 规范化可能改变了文本的断句和语义结构，split_paragraph 按标点分段后破坏了上下文连贯性。

## 问题 4: 整体合成质量差

**现象**: 即使用 `text_frontend=False` + 每段新建模型实例，合成的长文本音频仍存在:
- 丢字
- 语句不通畅
- 内容与原文不完全对应

**可能原因**: CosyVoice3 的 zero-shot 模式对长文本的上下文理解能力有限，分段后每段失去了前后文语境，导致生成内容与原文有偏差。

## 总结

CosyVoice3 在当前环境下不适合用于长文本有声书合成。主要瓶颈:
1. 模型推理状态污染，无法复用同一实例
2. 文本预处理会改变原始内容
3. 分段合成导致上下文丢失，内容与原文不对应
