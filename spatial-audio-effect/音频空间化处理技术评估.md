## 三、各组件详细配置对比

### 3.1 VoxCPM2（TTS 引擎）— 差异最大的组件

VoxCPM2 是当前默认 TTS provider，通过 subprocess 调用 CLI 二进制，不作为 Python 库 import。

| 维度 | Deepin 25 | Ubuntu 22.04/24.04 | Windows 10/11 | macOS |
|------|-----------|---------------------|---------------|-------|
| **安装方式** | `pip install voxcpm`（conda py312_env） | `pip install voxcpm`（venv/conda） | `pip install voxcpm`（有 CUDA 时） | 不支持 |
| **默认二进制路径** | `/home/muyi086/anaconda3/envs/py312_env/bin/voxcpm` | 需通过 `PATH` 或 `VOXCPM_BIN` 环境变量指定 | 需自行确认 | N/A |
| **默认模型路径** | `/persistent/home/muyi086/modelscope/VoxCPM2` | 需本地下载并通过 `AUDIO_3D_SDD_VOXCPM_MODEL_DIR` 指定 | 需本地下载 | N/A |
| **GPU 支持** | `--device auto`，自动检测 CUDA | `--device auto`，自动检测 CUDA | 有 CUDA 亦可 | N/A |
| **路径发现逻辑** | 优先硬编码路径 → fallback 到 `PATH` 搜索 | fallback 到 `PATH` 搜索 + 环境变量覆盖 | fallback 到 `PATH` 搜索 | N/A |
| **shlex 行为** | `posix=True` | `posix=True` | `posix=False`（`os.name == "nt"`） | `posix=True` |
| **批量优化** | 按 voice_id 分组避免模型重载 | 同左 | 同左 | N/A |

**关键配置要求**：

1. **二进制发现**：代码在 `tts_provider.py:25-26` 硬编码了 Deepin 路径作为默认值。在 Ubuntu 上，需要：
   - 方案 A：设置 `PATH` 使 `voxcpm` 可被 `shutil.which()` 发现
   - 方案 B：设置环境变量 `VOXCPM_BIN` 指向实际路径
   - 方案 C：修改 `DEEPIN_VOXCPM_BIN` 常量（不推荐，影响代码可维护性）

2. **模型目录**：代码在 `tts_provider.py:26` 和 `env_check.py:109` 硬编码了 Deepin 模型路径。在 Ubuntu 上，必须设置：
   ```bash
   export AUDIO_3D_SDD_VOXCPM_MODEL_DIR=/path/to/VoxCPM2
   ```

3. **CUDA 依赖**：VoxCPM2 默认 `--device auto`，会自动检测 GPU。Ubuntu 上需确保：
   - NVIDIA 驱动已安装
   - CUDA toolkit 版本与 VoxCPM2 兼容
   - `nvidia-smi` 可正常输出

4. **Windows 特殊处理**：`shlex.split()` 在 Windows 上使用 `posix=False` 模式，避免反斜杠被当作转义字符。这是代码中唯一的 Windows 特殊逻辑（`tts_provider.py:144,284`）。

## 六、建议

### 6.1 如果要在 Ubuntu 上运行完整生产流水线

1. 创建 conda 环境 `py312_env`，安装 Python 3.12
2. 安装 VoxCPM2：`pip install voxcpm`
3. 下载 VoxCPM2 模型到本地目录
4. 设置环境变量：
   ```bash
   export AUDIO_3D_SDD_VOXCPM_MODEL_DIR=/path/to/VoxCPM2
   export PATH="/path/to/voxcpm/bin:$PATH"
   ```
5. 安装 Ollama 并拉取 `qwen3.5:9b`
6. 安装 ffmpeg：`apt install ffmpeg` 或 `conda install ffmpeg`
7. 运行环境检查：`audio-3d-sdd check-env --tts-provider voxcpm`

### 6.2 代码改进建议

1. **路径配置化**：将 `DEEPIN_VOXCPM_BIN` 和 `DEEPIN_VOXCPM_MODEL` 改为优先读取环境变量，硬编码路径作为 fallback
2. **环境标识泛化**：`DEFAULT_REQUIRED_ENVIRONMENT` 应根据实际运行环境动态生成，而非硬编码 Deepin
3. **conda 环境名可配置**：`env_check.py` 中的环境名检测应支持环境变量覆盖
4. **`deepin_validation_required` 重命名**：考虑改为 `production_validation_required`，更准确表达语义

### 6.3 各系统推荐用途

| 系统 | 推荐用途 | 不推荐用途 |
|------|----------|------------|
| **Deepin 25** | 完整生产流水线、TTS 生成、QA、人工审听 | 无 |
| **Ubuntu 22.04/24.04** | 完整生产流水线（需配置路径）、CI/CD | 无 |
| **Windows 10/11** | 候选书筛选、文档编写、Schema 校验、元数据生成 | TTS 生成、GPU 推理 |
| **macOS** | 本地开发（非 TTS 部分）、测试 | TTS 生成、GPU 推理 |

## 八、结论

1. **Deepin 25 和 Ubuntu 在技术层面几乎完全兼容**，迁移成本极低（仅需调整 3 个路径/环境变量）
2. **VoxCPM2 是系统差异最大的组件**，但差异仅体现在路径发现和 GPU 检测，不涉及代码逻辑
3. **空间化后处理（`spatialize.py`）和音色匹配（timbre-design）完全跨平台**，无任何系统依赖
4. **Windows 仅适合作为辅助环境**，不适合运行完整生产流水线
5. **macOS 当前不支持 VoxCPM2**，不适合作为生产环境

### 9.2 难度分析

**Deepin 25（⭐ 最低）**：
- 代码硬编码了所有默认路径，开箱即用
- conda 环境、ffmpeg、Ollama 均已验证
- 硬件配置已确认：Intel 13600KF + 4070 Ti Super（16GB VRAM）+ 48GB RAM

**Ubuntu 22.04/24.04（⭐⭐ 低）**：
- 与 Deepin 同源（Debian），技术栈完全一致
- 唯一额外步骤：设置 2 个环境变量（`PATH` 和 `AUDIO_3D_SDD_VOXCPM_MODEL_DIR`）
- CUDA 驱动安装有完善文档（NVIDIA 官方支持 Ubuntu）

**Windows WSL2（⭐⭐⭐ 中）**：
- 需要先启用 WSL2 + 安装 Ubuntu 发行版
- CUDA 驱动装在 Windows 侧，WSL2 内通过 `libcuda.so` 透传
- 模型和数据必须放在 Linux 文件系统（`~/`），不能放 `/mnt/c`
- 配置步骤多一步，但安装完成后使用体验与 Ubuntu 一致

**Windows 原生（⭐⭐⭐⭐ 高）**：
- 需要手动安装 CUDA Toolkit、cuDNN、配置 PATH
- `shlex.split()` 需要 `posix=False` 模式（代码已处理）
- ffmpeg 需手动下载配置
- 部分 Python 包可能缺少 Windows 预编译 wheel
- 项目未测试此路径，可能遇到未知问题

**macOS（❌ 不可行）**：
- VoxCPM2 不支持 macOS
- 即使使用 CPU 模式，2B 模型推理速度也不可接受
- 仅适合做非 TTS 部分的开发和测试

| **Ubuntu（RTX 4090 24GB）** | ~3.0-3.5x 实时 | GPU compute | VRAM 更大，batch 更大 |
| **Ubuntu（CPU only）** | ~0.05-0.1x 实时 | CPU 算力 | 2B 模型 CPU 推理极慢，不实用 |
| **Windows WSL2（同 4070 Ti）** | ~2.3-2.4x 实时 | GPU compute | CUDA 透传，3-5% 开销 |
| **Windows 原生（同 4070 Ti）** | ~2.2-2.4x 实时 | GPU compute | CUDA 原生，但环境配置复杂 |
| **macOS（M2 Max）** | ❌ 不支持 | 模型不兼容 | VoxCPM2 无 macOS 支持 |

### 10.3 流畅度影响因素

影响 TTS 合成流畅度的关键因素，按影响程度排序：

| 因素 | 影响程度 | 说明 |
|------|----------|------|
| **GPU 型号** | ⭐⭐⭐⭐⭐ | 决定 CUDA 推理速度，2B 模型需要至少 8GB VRAM |
| **VRAM 容量** | ⭐⭐⭐⭐ | 决定 batch 大小和是否需要模型卸载重载 |
| **inference-timesteps** | ⭐⭐⭐ | 默认 4 步，增加会提高质量但降低速度 |
| **分段长度** | ⭐⭐ | 每段 80-500 字，段数越多 overhead 越高 |
| **文件系统 I/O** | ⭐⭐ | 模型加载和 WAV 写入，WSL2 中放 `/mnt/c` 会慢 2-3x |
| **系统差异** | ⭐ | Deepin/Ubuntu 几乎无差异，WSL2 仅 3-5% 开销 |

## 十一、Windows WSL2 专题分析

### 11.1 Windows 能否原生运行 VoxCPM2？

**技术上可以，但不推荐**。原因：

1. VoxCPM2 是 Python 包（`pip install voxcpm`），Windows 上理论上可安装
2. 但项目**从未在 Windows 原生环境测试过** TTS 生成
3. 代码中 `os.name != "nt"` 的 shlex 处理表明开发者考虑过 Windows，但仅限于命令解析
4. CUDA 在 Windows 上的配置比 Linux 复杂得多（cuDNN、PATH、版本匹配）
5. 部分依赖可能缺少 Windows 预编译 wheel

### 11.2 WSL2 是什么？为什么说"Windows 只能跑 WSL"？

这个说法**部分正确**：

- **非 TTS 部分**（筛选、文本清洗、Schema 校验、元数据）：Windows 原生可跑，不需要 WSL
- **TTS 部分**（VoxCPM2 GPU 推理）：Windows 原生技术上可跑，但 WSL2 是更现实的路径

WSL2 的优势：
- 提供完整的 Linux 环境，与 Ubuntu 行为一致
- CUDA 通过 `libcuda.so` 透传到 Windows 驱动，GPU 推理接近原生
- 安装 `voxcpm`、`ffmpeg`、`conda` 等与 Linux 完全一致
- 不需要折腾 Windows 版 CUDA/cuDNN

### 11.3 WSL2 的性能损耗

| 方面 | 损耗 | 说明 |
|------|------|------|
| **GPU 推理** | **~1-3%** | CUDA 计算通过驱动透传，几乎无损耗 |
| **模型加载** | **~5-20%** | 取决于文件位置；放 Linux 文件系统（`~/`）几乎无损耗 |
| **文件 I/O（/mnt/c）** | **~100-200%** | 跨文件系统访问 Windows 盘，**非常慢** |
| **文件 I/O（Linux fs）** | **~0-5%** | 原生 ext4，与 Linux 一致 |
| **内存** | **+200-500MB** | WSL2 虚拟机本身的固定开销 |
| **启动** | **+2-5 秒** | WSL2 冷启动时间 |
| **端到端 TTS 流水线** | **~3-5%** | 假设模型和数据放在 Linux 文件系统 |

**关键结论**：WSL2 的性能损耗**可以忽略不计**，前提是：
1. 模型和数据放在 Linux 文件系统（`~/`），不要放 `/mnt/c`
2. 使用最新的 NVIDIA WSL2 CUDA 驱动
3. 不要在 WSL2 中使用 Docker 嵌套（会增加额外开销）

### 11.4 WSL2 的已知限制

| 限制 | 影响 | 规避方式 |
|------|------|----------|
| 多 GPU 支持有限 | 本项目单 GPU 无影响 | 不适用 |
| 部分 CUDA profiling 工具不可用 | 不影响 TTS 生产 | 仅影响开发调试 |
| 文件系统跨边界慢 | 放 `/mnt/c` 会慢 | 数据放 `~/` |
| Docker + WSL2 + CUDA 嵌套 | 额外 5-10% 开销 | 不用 Docker |
| `nvidia-smi` 部分功能受限 | 不影响推理 | 仅影响高级管理 |

## 十二、最终推荐

### 推荐方案：Deepin 25 > Ubuntu > Windows WSL2

| 排名 | 系统 | 推荐理由 | 适用场景 |
|------|------|----------|----------|
| 🥇 | **Deepin 25** | 已验证、零配置、2.4x 实时、代码默认支持 | 当前生产环境，继续使用 |
| 🥈 | **Ubuntu 22.04/24.04** | 与 Deepin 同源、仅需 2 个环境变量、CI/CD 友好 | 新机器、服务器、CI 环境 |
| 🥉 | **Windows WSL2** | 损耗仅 3-5%、CUDA 透传成熟、安装步骤多一步 | 只有 Windows 机器时的替代方案 |
| 4 | **Windows 原生** | 未测试、配置复杂、可能有未知问题 | 仅非 TTS 部分（筛选/文档） |
| ❌ | **macOS** | VoxCPM2 不支持 | 不适用 |

### 具体建议

**如果你现在有 Deepin 25 机器**：
→ 继续用，不需要改任何东西。这是最佳选择。

**如果你需要在新机器上部署**：
→ 优先装 Ubuntu 22.04/24.04，配置成本极低，只需：
```bash
# 1. 安装 conda + 创建环境
wget https://repo.anaconda.com/miniconda/Miniconda3-latest-Linux-x86_64.sh
bash Miniconda3-latest-Linux-x86_64.sh
conda create -n py312_env python=3.12
conda activate py312_env

# 2. 安装依赖
pip install voxcpm
conda install ffmpeg
pip install -e .

# 3. 设置环境变量
export AUDIO_3D_SDD_VOXCPM_MODEL_DIR=/path/to/VoxCPM2
export PATH="/path/to/voxcpm/bin:$PATH"

# 4. 安装 Ollama
curl -fsSL https://ollama.com/install.sh | sh
ollama pull qwen3.5:9b

# 5. 验证
audio-3d-sdd check-env --tts-provider voxcpm --expected-conda-env py312_env --require-pytest
```

**如果你只有 Windows 机器**：
→ 方案 A（推荐）：安装 WSL2 + Ubuntu，性能损耗可忽略
```powershell
# Windows PowerShell 管理员
wsl --install -d Ubuntu-24.04
# 然后进入 WSL，按 Ubuntu 步骤操作
```
→ 方案 B（仅非 TTS）：Windows 原生做筛选/文档/校验，TTS 部分标记 `unverified`，后续在 Deepin/Ubuntu 复跑

**关于 WSL2 性能损耗的结论**：
→ GPU 推理损耗仅 **1-3%**，端到端 TTS 流水线损耗约 **3-5%**
→ 前提是模型和数据放在 Linux 文件系统（`~/`），不放 `/mnt/c`
→ 对于 26 分钟样章，10.9 分钟变成 ~11.2 分钟，差异可忽略
→ **不是"只能跑 WSL"，而是 WSL 是 Windows 上跑 TTS 的最优路径**
